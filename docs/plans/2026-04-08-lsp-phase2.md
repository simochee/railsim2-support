# LSP Phase 2: Semantic Schema + Schema Validator 実装計画

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** RailSim2 定義ファイルの意味レベルのエラー（無効なプロパティ、型不一致、必須プロパティ欠落）をエディタ上で表示する

**Architecture:** `src/schema/semantic.ts` に全オブジェクトの構造定義（プロパティ型、子オブジェクト、必須性）を持ち、`schemaValidator.ts` が AST を走査してスキーマと突き合わせる。ファイル種別ごとのトップレベル構造は `fileSchemas: FileSchema` で定義し、これがルート検証の唯一の情報源。`ObjectSchema` には `rootFor` を持たせない（二重管理防止）。parser は引き続き keywords.ts 非依存。

**Tech Stack:** TypeScript (tsc), vitest

**設計判断:**
- semantic schema はヘルプドキュメントから手動で抽出したデータを TypeScript で記述
- `Vertex` のように親によって型が異なるケース: validator が**親オブジェクト名を引数に受け取り**、`ObjectSchema.children` で定義された子マッピングを辿って正しいスキーマ定義を解決する。schema データ上は `Face` の子に `Vertex` → schema key `"Vertex:Profile"` のように `親コンテキスト:名前` でルックアップする
- 数値式の型チェック: `float`/`integer` 判定では、`number` リテラルだけでなく `unary("-", number)` も許容する。式 (`binary` ノード) が出現した場合は `expression` 型として扱い型エラーにしない
- ルート情報は `fileSchemas: FileSchema` に一本化: `semantic.ts` 内で `fileSchemas` を定義し、`getFileSchema(fileName)` でルート定義を返す。`ObjectSchema` に `rootFor` は持たせない
- `validateTextDocument(text, fileName?)` — fileName はオプショナル。省略時はルート検証をスキップ（既存テスト後方互換）

### Task 1: schema 型定義 (`src/schema/schemaTypes.ts`)

**Files:**
- Create: `src/schema/schemaTypes.ts`

型定義のみ。

```ts
export type PropertyType =
  | "float" | "integer" | "string" | "filename" | "identifier"
  | "yes-no" | "color" | "expression"
  | "vector-2d" | "vector-3d"
  | "enum";

export interface PropertySchema {
  type: PropertyType;
  required: boolean;
  multiple: boolean;         // true = 同名プロパティ複数回許可
  arity?: number;            // 値の個数。省略時は1。Coord = 1.0, 2.0; なら arity: 2
  enumValues?: string[];     // type === "enum" の場合
}

export interface ChildSchema {
  required: boolean;
  multiple: boolean;          // true = 複数配置可
  schemaKey?: string;         // 親コンテキスト依存の場合の lookup key（例: "Vertex:Profile"）
}

export interface ObjectSchema {
  properties: Record<string, PropertySchema>;
  children: Record<string, ChildSchema>;
  nameParameter?: PropertyType;  // Object3D "name" の引数型（Phase 2 では型のみ定義、検証は将来対応）
}
// 注意: rootFor は持たない。ルート情報は fileSchemas (FileSchema) が唯一の情報源。

export type SemanticSchema = Record<string, ObjectSchema>;

/** ファイルレベルのルートオブジェクト定義 */
export interface RootObjectEntry {
  name: string;
  required: boolean;
  multiple: boolean;
}

/** ファイル種別ごとのトップレベル構造 */
export type FileSchema = Record<string, RootObjectEntry[]>;
```

/** ファイル種別ごとのトップレベル構造を定義 */
export const fileSchemas: FileSchema = {
  "Rail2.txt": [
    { name: "PluginHeader", required: true,  multiple: false },
    { name: "RailInfo",     required: true,  multiple: false },
    { name: "SoundInfo",    required: true,  multiple: false },
    { name: "Profile",      required: false, multiple: true },
    { name: "Wireframe",    required: false, multiple: true },
    { name: "Interval",     required: false, multiple: true },
  ],
  // ... 12ファイル分
};

/** ファイル名からルートオブジェクト定義を返す */
export function getFileSchema(fileName: string): RootObjectEntry[] | undefined {
  return fileSchemas[fileName];
}
```

```

### Task 3: Schema validator (`src/server/validator/schemaValidator.ts`)

**Files:**
- Create: `src/server/validator/schemaValidator.ts`
- Create: `test/schemaValidator.test.ts`

AST + スキーマ + ファイル名から意味レベルのエラーを検出。

**validator の走査方式:**
validator は AST を再帰的に走査する。各 ObjectNode を訪問する際:
1. 現在のオブジェクトの schema を lookup（`schemaKey` があればそれを使う、なければ `node.name` で引く）
2. schema が見つからなければスキップ（unknownKeywordValidator の責務）
3. 見つかれば以下を検証

**検証内容:**
1. **無効なプロパティ**: PropertyNode.name が schema.properties にない → error
2. **型不一致**: プロパティ値の ExprNode 型が期待と異なる → error
3. **必須プロパティ欠落**: schema.properties で required=true なのに body にない → warning
4. **値の arity 不一致**: arity が定義されている場合、値の個数が一致しない → error
5. **無効な子オブジェクト**: ObjectNode.name が schema.children にない → error
6. **ファイルレベル root 検証**: getFileSchema(fileName) でルート定義を取得し:
   - トップレベルの ObjectNode がルート定義に含まれるか → error (不正なルートオブジェクト)
   - required=true のルートオブジェクトが存在するか → warning (必須ルート欠落)
   - multiple=false のルートオブジェクトが2回以上出現していないか → error (重複)
   - トップレベルの IfNode / ApplySwitchNode はルート検証をスキップする（条件分岐の中身まで追う必要はない。Phase 2 ではトップレベル ObjectNode のみ検証対象）

**注意:** `keywords.ts` のパスは `src/shared/keywords.ts`（`src/schema/` ではない）。

**型チェックロジック:**
- `float`: number リテラル、または unary("-", number)。binary 式は `expression` 扱いでスキップ
- `integer`: float と同様 + 小数部がないこと
- `string` / `filename`: string リテラル
- `yes-no`: boolean リテラル (`yes`/`no`)
- `color`: color リテラル
- `identifier`: identifier リテラル
- `vector-2d`: 値が2個で各要素が float 互換
- `vector-3d`: 値が3個で各要素が float 互換
- `enum`: identifier かつ enumValues に含まれること
- `expression`: 任意（チェックなし）

**テスト:**
- 有効な入力 → エラーなし
- 無効プロパティ → error
- 型不一致: string を float に → error
- 型不一致: float を string に → error
- 負数 `-1.0` が float として許容される
- 必須プロパティ欠落 → warning
- 値 arity 不一致 (vector-3d に値2個) → error
- 無効な子オブジェクト → error
- ファイルレベルのルート検証 (Rail2.txt に TrainInfo → error)
- ネストしたオブジェクトの検証
- スキーマ未定義オブジェクトはスキップ
- 親コンテキスト依存 (Profile内 Vertex vs Wireframe内 Vertex)
- fileName 省略時はルート検証スキップ
- yes-no 型に数値 → error

**Step 1-3:** テスト → 実装 → 通過確認 → コミット
